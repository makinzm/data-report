# Analysis Definition of Done (DoD)

データ分析プロジェクトの完了基準。
すべての分析はこの DoD を満たすことでレビュー可能・公開可能とみなす。

---

## 0. 前提：HARKing 禁止

**HARKing（Hypothesizing After Results are Known）を禁止する。**

- 分析を始める前に仮説・リサーチクエスチョンを必ず文書化する
- 結果を見てから「最初からそう思っていた」と書き換えない
- 探索的分析（EDA）と検証的分析（CDA）を明確に分離する
  - EDA：パターン発見目的。仮説生成フェーズ
  - CDA：事前に立てた仮説を検証するフェーズ
- EDA で発見したパターンを同じデータで「検証」しない

---

## 1. 分析着手前チェックリスト

着手前に `analysis/<YYYYMMDD>-<title>/PROPOSAL.md` を作成し、以下をすべて埋める。

### 1.1 リサーチクエスチョン
- [ ] 分析で答えたい問いを1〜2文で明示している
- [ ] 問いは「なぜ」「どれくらい」「どちらが」など、答えられる形になっている
- [ ] 「手法を試したい」「なんとなく面白そう」が動機になっていない

### 1.2 動機・背景
- [ ] この問いに答えることで誰が何を得るか書いている
- [ ] 社会的・実務的意義、または学習上の意義を書いている

### 1.3 仮説（検証的分析の場合）
- [ ] 結果を見る前に仮説を明文化している
- [ ] 仮説は反証可能な形（「〜だと予想する。なぜなら〜だから」）になっている

### 1.4 スコープ
- [ ] 分析対象の期間・地域・集団などを明示している
- [ ] 答えない問い・対象外事項も書いている（スコープ外の明示）

---

## 2. データチェックリスト

### 2.1 データソース
- [ ] データの出典（URL・バージョン・取得日時）を記録している
- [ ] ライセンス・利用規約を確認し記載している
- [ ] データの粒度・対象期間・件数を記載している

### 2.2 データ品質
- [ ] 欠損値の件数・割合・対処方法を記録している
- [ ] 外れ値の定義と対処方法を記録している
- [ ] データの限界（バイアス・代表性の問題など）を記載している

### 2.3 再現性
- [ ] データ取得スクリプト、または取得手順を文書化している
- [ ] シード値など乱数の固定を行っている（該当する場合）
- [ ] 環境情報（Python バージョン・主要ライブラリバージョン）を記録している（`requirements.txt` or `pyproject.toml`）

---

## 3. 分析手法の選定記録（ADR）

`analysis/<YYYYMMDD>-<title>/decisions/` 以下に手法選定を記録する。  
1 つの判断につき 1 ファイル（例：`001-why-logistic-regression.md`）。

```markdown
# ADR-NNN: <手法名・判断内容>

## ステータス
Accepted / Superseded by ADR-NNN / Deprecated

## コンテキスト
- 何を達成したいか
- 当時の状況・制約（データ量、変数の型、解釈性の要求 など）

## 選択肢
| 手法 | メリット | デメリット |
|------|---------|-----------|
| A    | ...     | ...       |
| B    | ...     | ...       |

## 決定
<選んだ手法> を選ぶ。

## 理由
- ...
- ...

## トレードオフ・リスク
- ...

## 結果（実施後に記入）
- 想定通りだった点：
- 想定外だった点：
```

チェック：
- [ ] 採用した統計・ML 手法ごとに ADR を作成している
- [ ] 「とりあえずやってみた」だけの手法選定になっていない
- [ ] 前提条件（正規性・独立性など）の確認結果を記載している

---

## 4. 分析実施チェックリスト

### 4.1 探索的分析（EDA）
- [ ] 各変数の基本統計量（平均・中央値・分散・分布形状）を確認している
- [ ] 可視化で外れ値・分布・時系列トレンドを確認している
- [ ] EDA の発見は「発見事項」として記録し、仮説と混在させていない

### 4.2 検証的分析（CDA）
- [ ] 事前に定義した仮説に対して検定・モデリングを行っている
- [ ] 有意水準・検出力・効果量を事前に設定している（該当する場合）
- [ ] 多重検定を行う場合は補正方法（Bonferroni 等）を明記している
- [ ] 統計的有意性と実務的意味の違いを区別して説明している

### 4.3 可視化
- [ ] 軸ラベル・タイトル・凡例が揃っている
- [ ] グラフの選択理由が明確（棒グラフ vs 折れ線 など）
- [ ] 誤解を招く y 軸の切り捨てをしていない

---

## 5. 結論・レポートチェックリスト

### 5.1 論理構成
- [ ] リサーチクエスチョンへの直接的な回答を最初に書いている
- [ ] 結論は分析結果から導いている（結果に反する結論を書いていない）
- [ ] 相関と因果を混同していない（「〜の傾向がある」と「〜が原因だ」を区別）

### 5.2 限界と留意事項
- [ ] 分析の限界（データの偏り・手法の仮定が破られている場合等）を記載している
- [ ] 結果を一般化できる範囲を明示している
- [ ] 今後の分析・改善案を記載している

### 5.3 再現性の最終確認
- [ ] クリーンな環境でノートブックを最初から最後まで実行して結果が再現できた
- [ ] コードに README または実行手順を記載している

---

## 6. レビュー基準

レビュワー（または DA レビュワー Agent）は以下の観点で確認する。

| 観点 | 確認内容 |
|------|---------|
| HARKing | 仮説が結果前に文書化されているか |
| 目的の明確さ | RQ が答えられる形か、動機が明確か |
| 手法の妥当性 | ADR が書かれており、前提条件を確認しているか |
| 再現性 | クリーン実行で結果が一致するか |
| 論理の整合性 | 結論が結果から導かれているか、相関/因果を混同していないか |
| 限界の記載 | 過剰な一般化をしていないか |

---

## 7. ディレクトリ構成（テンプレート）

```
analysis/
└── YYYYMMDD-<title>/
    ├── PROPOSAL.md          # 着手前チェックリスト（RQ・仮説・スコープ）
    ├── README.md            # 分析の概要・実行手順
    ├── decisions/
    │   └── 001-<手法名>.md  # ADR
    ├── data/
    │   ├── raw/             # 取得したままのデータ（変更しない）
    │   └── processed/       # 加工済みデータ
    ├── notebooks/
    │   ├── 01_eda.ipynb     # 探索的分析
    │   └── 02_analysis.ipynb # 検証的分析
    ├── outputs/
    │   └── figures/         # 出力グラフ
    └── REPORT.md            # 最終レポート
```
