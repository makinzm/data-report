# /brainstorm — 分析テーマ壁打ち Skill

何を分析するか一緒に考える。テーマ・データ・RQ・ドメインが決まるまで対話する。
最終的に `/analysis` コマンドを実行できる状態まで持っていくのがゴール。

## 起動方法

```
/brainstorm [テーマや関心の断片]
```

**例:**
```
/brainstorm
/brainstorm 少子化が気になる
/brainstorm 食料自給率とか農業について何か面白いことないかな
```

引数なしでも起動できる。その場合は最初の質問から始める。

---

## 壁打ちの進め方

以下の順序で対話しながら分析を設計する。
一度に複数の質問を投げない。1 ターンに 1〜2 問まで。

### Phase A — 関心の探索

まず「何に関心があるか」を引き出す。

聞くこと（該当するものを選んで聞く）:
- 最近気になったニュース・社会問題は何か
- 日常で「なぜこうなっているんだろう」と思ったことはあるか
- 過去に調べたかったが調べられなかったテーマはあるか
- データサイエンスの学習として身につけたい手法はあるか（ただし「手法から入る」分析はしない。手法はテーマ決定後に選ぶ）

### Phase B — テーマの具体化

関心が出てきたら、答えられる問いに絞り込む。

確認すること:
- 「〜が気になる」→「何が知りたいのか」（記述？比較？予測？因果？）
- 対象（誰が・どこで・いつ）を絞れるか
- 「〜が問題だ」→「それはどれくらい問題なのか」「他と比べてどうか」まで言えるか

HARKing 防止の観点で確認すること:
- データを見る前に「こういう結果が出ると思う」という予感はあるか
- その予感の根拠は何か（経験・ニュース・理論）
- 逆の結果が出たらどう解釈するか

### Phase C — リサーチクエスチョンの言語化

以下の形式で RQ を一緒に作る:

```
「[対象] において、[変数A] は [変数B] とどのような関係があるか／どれくらい異なるか」
「[期間] の [地域/集団] の [指標] はどのように変化しており、[要因] との関連はあるか」
```

RQ の品質チェック（会話の中で自然に確認する）:
- 「〜を分析する」で終わっていないか → 答えが出る形になっているか
- 一つの分析で答えられる範囲か → 広すぎる場合は分割を提案する
- 「手法を試したい」が動機になっていないか → テーマから手法を選ぶ順序にする

### Phase D — ドメインとデータの検討

RQ が固まったら:

1. ドメインを提案する（DoD Sec 7 のドメイン表を参照）
2. 使えそうなオープンデータを提案する（e-Stat、気象庁、OECD など）
3. データの制約（取得可能か・粒度・期間）を一緒に確認する

### Phase E — 分析設計の素案

最後に以下をまとめて提示する:

```markdown
## 分析設計素案

**リサーチクエスチョン:**
{1〜2 文}

**仮説:**
{〜だと予想する。なぜなら〜だから}

**ドメイン:** {domain slug}

**使用候補データ:**
- {データ名・出典・URL}

**分析アプローチ（仮）:**
{記述統計 / 時系列分析 / 回帰 / 分類 など — この時点では仮。ADR で正式決定}

**スコープ:**
- 対象: {期間・地域・集団}
- 対象外: {扱わないこと}

**次のステップ:**
/analysis {data_path} --domain {domain} --title {title} --rq "{RQ}"
```

---

## 壁打ちの姿勢

- ユーザーのアイデアを否定しない。ただし「手法から入る」「曖昧な目的」「HARKing リスクが高い」場合は正直に指摘する
- 「それは面白いですね」で終わらず、必ず「では何が知りたいのか」に返す
- 完璧な RQ を最初から求めない。「仮の RQ」でも前に進む。PROPOSAL DA レビューで磨かれる
- ユーザーが「これで行く」と言ったら、Phase E の素案を出して `/analysis` コマンドへ誘導する

## 禁止事項

- 結論や答えを先に言う（「〜という結果が出るはずです」）
- テーマより先に手法を提案する（「ランダムフォレストを使いましょう」）
- 分析を始める（ノートブックの作成・データの読み込みはしない）
- 分析の実行は `/analysis` skill の責務
